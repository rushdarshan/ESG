# In-depth Project README

## Overview

Replace the single-line `README.md` ("# EcoSphere - ESG Management Platform") with a comprehensive, in-depth project README that serves as the canonical entry point for anyone encountering the project — contributors, reviewers, hackathon judges, or future maintainers.

The README should communicate what EcoSphere is, why it exists, how it's built, how to run it, and how to contribute — all grounded in the actual implemented state of the codebase.

---

## Problem Frame

The current `README.md` at the repo root is a single heading with no content. For a hackathon project with 3 contributors, 14 database models, 10 pages, 6 API routes, and 12 library modules, this is insufficient. An in-depth README provides:

- Immediate project understanding for judges/reviewers
- Onboarding guidance for new contributors
- Architecture documentation that lives alongside the code
- Setup instructions that actually work

## Scope Boundaries

### In scope

- Replace `README.md` at repo root with comprehensive content
- All content derived from actual codebase state (not aspirational)
- Sections ordered for progressive disclosure: overview → setup → architecture → features → API → contributing

### Out of scope

- API documentation site (separate concern)
- Inline code documentation / JSDoc
- Diagrams or images (text-only README)
- Translation / multi-language README

---

## Approach

Write the README in a single file at `README.md` (repo root), replacing the existing single-line content. Structure follows standard open-source README conventions with ESG-specific sections.

### README Structure

1. **Title + tagline** — "EcoSphere — Unified E+S+G Enterprise Sustainability Platform"
2. **One-paragraph overview** — what it is, who it's for, the core value proposition
3. **Key features** — bullet list organized by pillar
4. **Tech stack** — table with layer, technology, version, purpose
5. **Getting started**
   - Prerequisites (Node.js, PostgreSQL, npm)
   - Clone + install
   - Environment variables (reference `.env.example`)
   - Database setup (prisma generate, prisma db push, prisma db seed)
   - Running dev server
6. **Project structure** — tree with annotations for key directories
7. **Architecture**
   - Data model overview (14 Prisma models, key relationships)
   - Key design decisions (monorepo, AI provider interface, evidence hash chain, MACC knowledge base)
8. **Features deep dive**
   - Environmental: upload + AI extraction, scope 1/2/3 calculation, MACC curve, provenance chain
   - Social: employee actions, XP/levels/badges, leaderboards
   - Governance: GRI/CSRD compliance, evidence registry, audit logs
   - Cross-cutting: AI assistant (Gemini), PDF reports, demo scenarios
9. **API endpoints** — table with route, method, description
10. **Testing** — run command, test file listing, coverage summary
11. **Demo data** — GreenForge scenarios explanation
12. **Contributing** — branch naming, PR process, code style (Tailwind, TypeScript strict)
13. **Current status** — what's working, what's static/hardcoded, known limitations
14. **License** — MIT or appropriate

## Risks

- **Staleness risk:** README content is derived from current code state; as code changes, README may drift. Mitigated by keeping descriptions tied to implementation, not plans.
- **Accuracy risk:** Factual errors in setup instructions. Mitigated by cross-referencing with package.json scripts and .env.example.

---

## Decisions

- **Single file:** All content in one `README.md` rather than splitting into docs/ — appropriate for project size and hackathon context
- **No diagrams:** Text-only for portability and ease of maintenance
- **Status section included:** Explicitly called out what's static vs live — important for hackathon judges to understand the prototype state
